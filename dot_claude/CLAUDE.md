# 開発指針

一貫性、可読性、保守性を重視しシンプルな実装を心がける。
不明瞭な指示は質問を行い明確にする。
公式ドキュメント等の1次ソースをまず調査する。
非自明な変更では、実装に着手する前にインターフェース仕様（シグネチャ・型・エラー仕様など）を確定する。

# レビュー・テスト方針

テストは実装をなぞらず仕様・インターフェースに対して書くものであり、実装完了を待つ必要はない。インターフェース仕様確定後は、メイン実装とテスト実装を並行して進める。`test-author` には仕様と制約を渡し、Agent ツールを `run_in_background: true` で起動してメイン実装をブロックしない。
`code-reviewer` によるレビューは実装とテストの両方が完成してからまとめて行う（レビューは実装完了に依存するため並行不可）。自己正当化バイアスを排除するため、実装を書いたコンテキストとは別エージェントに行わせる。
小さな変更・自明な修正では上記を省略してよい（オーバーヘッド回避）。
レビュアー/テスト担当には仕様・変更の意図・差分・制約を必ず渡す（隔離コンテキストには要約しか伝わらないため）。
`code-reviewer` のレビュー結果は会話に提示する（サブエージェントの返り値は tool result として親に返るだけでユーザーの画面には出ず、Ctrl+O でも展開されないため）。
